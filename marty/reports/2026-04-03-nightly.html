4. Deep Dive: QA Strategy Without a Dedicated QA Person

Topic from Marty's 3pm suggestion — no response from Jake, so going with this one.

What It Is

A QA strategy for early-stage startups that don't have (and can't justify) a full-time QA engineer. Instead of one person owning all testing, you distribute QA responsibility across the team with lightweight processes, automation, and clear ownership of "done" criteria. The goal: ship confidently without a gatekeeper bottleneck.

Why It Matters Right Now

Andreas left at the end of March. 22+ tasks are stuck in "for testing" with no QA owner. The team shipped 11 PRs this week, but zero tasks were marked "done" — everything piles up in the testing column. Warda is onboarding as AI QA, but the gap is real today. Without a strategy, the TestFlight build for customer interviews will have unknown quality.

Key Takeaways

  • Make "done" mean "tested by someone who didn't write the code." Every PR that merges should have a QA checklist: the dev writes what to test, another person (or Warda) runs it. No exceptions.
  • Adopt a "QA buddy" system. Nadeem tests Asif's work, Asif tests Rizwan's, Rizwan tests Nadeem's. Rotate weekly. Each dev owns 30 minutes of testing per day — it's part of the job, not extra.
  • Automate the critical path first. The invite flow (phone → OTP → join trip) and the iMessage → app sync should have automated smoke tests. Playwright or Maestro (mobile) can cover these in under a day of setup. Startup QA tools guide
  • Close the ClickUp loop. Right now "for testing" is a graveyard. Define: if a task sits in "for testing" for 48 hours, the original dev is responsible for self-testing and moving it to done. Stale = owned by author.
  • Warda's role: triage, not gatekeep. AI QA should focus on exploratory testing (finding new bugs), not blocking releases. Use her to stress-test edge cases the devs won't think of — multi-user, offline, weird phone numbers, emoji-heavy inputs.

One Action for Monday

Have Asif define a QA handoff template in ClickUp: when a task moves to "for testing," the dev must fill in (1) what changed, (2) how to test it, (3) what could break. If those fields are empty, it bounces back to "in progress."

11. CS 101 — Jake's Daily Lesson

What Is DNS? (And What Happens When Someone Types jointryps.com)

When you type jointryps.com into a browser, your computer has no idea where that is. It's like saying "take me to Jake's apartment" — you need an address first. DNS (Domain Name System) is the internet's phone book. It translates human-readable names like jointryps.com into IP addresses like 76.76.21.93 — the actual "street address" of the server where Tryps lives.

Here's what happens in ~100 milliseconds when someone opens Tryps:

  1. Browser asks your Wi-Fi router: "Where is jointryps.com?"
  2. Router doesn't know, so it asks your ISP's DNS server (like Verizon or Google's 8.8.8.8)
  3. ISP asks the .com registry: "Who handles jointryps.com?"
  4. Registry says: "Ask Vercel's nameservers" (because that's where Tryps' DNS is configured)
  5. Vercel's nameserver responds: "jointryps.com lives at 76.76.21.93"
  6. Browser connects to that IP → Tryps loads

Why it matters for Tryps: When Nadeem's PR #333 consolidated all jointripful.com references to jointryps.com, DNS is what makes sure both domains eventually reach the same server. The deep link scheme (tripful://) works differently — that's handled by the phone's OS, not DNS — but the web landing pages, OG images, and the invite link all depend on DNS resolving correctly. If DNS is misconfigured, every invite link is broken.

The "aha" moment: DNS is why a domain costs money — you're paying for a permanent entry in the internet's phone book that says "this name belongs to this server." Change your server? Just update the phone book entry. That's what a "DNS propagation" is: waiting for every copy of the phone book worldwide to get the update.
