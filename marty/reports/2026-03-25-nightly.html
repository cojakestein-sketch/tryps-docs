4. Deep Dive: The LiteLLM Supply Chain Attack

Topic selected by Marty — couldn't read Jake's DM response to the 3pm suggestion (Slack channel_not_found error). This is today's biggest security story and directly relevant to our stack.

What Happened

On March 24, a threat group called TeamPCP compromised the PyPI packages for LiteLLM (versions 1.82.7 and 1.82.8) — a widely-used Python SDK that proxies 100+ LLM APIs. The attack exploited a prior compromise of the Trivy security scanner's CI/CD pipeline, which gave attackers access to the LiteLLM CEO's GitHub account. The malicious packages contained a credential stealer and malware dropper targeting cloud credentials, SSH keys, Kubernetes configs, CI/CD secrets, crypto wallets, and shell history.

Why This Matters for Tryps Right Now

We don't use LiteLLM directly, but we use the same open-source supply chain (npm, PyPI). Our agent-intelligence scope (Rizwan) is building infra that will call multiple LLM providers. This attack demonstrates that even "trusted" packages with millions of downloads can be silently compromised through cascading CI/CD breaches. The transitive risk — where a compromised security tool (Trivy) leads to a compromised LLM tool (LiteLLM) — is the pattern to worry about.

Key Takeaways

  • Pin dependencies: Never use floating versions in production. Lock exact versions in package.json and requirements.txt. This attack only hit 1.82.7-1.82.8 — pinned users on 1.82.6 were safe.
  • Audit CI/CD secrets: The attack chain started with compromised CI credentials. Review GitHub Actions secrets, Supabase tokens, and EAS build credentials. Rotate anything that's been in plaintext.
  • Monitor for compromised packages: Set up automated alerts (e.g., Socket.dev, npm audit) on all production dependencies. Especially anything that handles credentials or API keys.
  • Isolate agent execution: As we build the Tryps agent, run LLM calls in sandboxed environments with minimal credential exposure. Don't give the agent process access to the full secret store.
  • Action item: Have Rizwan add a dependency audit step to the Maestro/QA process he's setting up. 10 minutes of setup, could save a catastrophic breach.
