4a. CI/CD — Continuous Integration / Continuous Deployment

What it is

CI/CD is an automated assembly line for code. When a developer pushes code to GitHub, a robot (called a "runner") grabs that code, runs every test in the codebase, checks for lint errors, and — if everything passes — automatically builds the app and deploys it. "Continuous Integration" means every code change gets tested immediately when it's pushed. "Continuous Deployment" means passing code gets shipped to users without a human pressing a button.

How it works at Tryps

When Nadeem opens a PR on GitHub, GitHub Actions (our CI) kicks off. It installs all 560+ npm packages, runs our 1,379 tests, and checks TypeScript types. If anything fails, the PR gets a red ❌ and can't be merged. When the PR is merged to develop, a separate workflow could (in theory) run eas update to push an OTA update to every phone running Tryps.

Right now, Tryps has CI (tests run on PRs) but not full CD (deploys are still manual via eas build and eas update). That's normal for a pre-launch app — you want a human in the loop before shipping to TestFlight.

The "aha" moment

CI/CD is the reason you can trust a merge. Without it, merging code is like crossing your fingers. With it, a robot has already verified the code works before it ever touches the main branch.

Key terms

  • Pipeline: The sequence of steps (install → test → build → deploy)
  • Runner: The computer that executes the pipeline (GitHub gives you free ones)
  • Green build: All tests passed — safe to merge
  • Red build: Something broke — PR is blocked
  • OTA update: Push a code change to users' phones without going through the App Store (Expo's eas update)

4b. SDKs — Software Development Kits

What it is

An SDK is a toolbox that someone else built so you don't have to. When Supabase gives you @supabase/supabase-js, that's an SDK — it contains pre-built functions like supabase.auth.signIn() and supabase.from('trips').select(). Instead of writing raw HTTP requests to Supabase's servers, you call these neat little functions. The SDK handles the ugly networking stuff under the hood.

SDKs Tryps uses right now

  • Expo SDK 54: The big one. Gives us camera, push notifications, location, haptics, file system access — basically everything that makes a phone app feel like a phone app instead of a website.
  • Supabase JS SDK: Database queries, authentication, realtime subscriptions, file uploads
  • Stripe SDK: Payment processing (coming in P2)
  • Duffel SDK: Flight search and booking (coming in P3)
  • ElevenLabs SDK: Text-to-speech for the agent voice

The "aha" moment

An SDK is like hiring someone who already knows how to talk to a specific service. You say "sign this user in," and the SDK translates that into the exact HTTP requests, headers, tokens, and error handling that the service expects. Without SDKs, every developer would have to read 200-page API docs and write boilerplate networking code for every single service they integrate.

SDK vs API

An API is the restaurant's menu — it tells you what you can order. An SDK is the waiter — it takes your order, brings it to the kitchen, and brings back the food. You could walk into the kitchen yourself (raw API calls), but the waiter (SDK) makes life much easier.

4c. CLIs vs MCPs — Two Ways Machines Talk to Services

CLI: Command Line Interface

A CLI is a program you type commands into. When Asif types gh pr list, he's using the GitHub CLI. When you type eas build, that's the Expo CLI. CLIs are designed for humans — you read the output on screen, decide what to do next, and type the next command.

MCP: Model Context Protocol

MCP is a new standard (Anthropic created it) that lets AI agents talk to services the same way CLIs let humans talk to services — but structured for machines, not humans. Instead of an AI reading text output and parsing it (fragile), MCP sends and receives structured JSON data with clear schemas.

The key difference

Why this matters for Tryps

Marty uses MCPs to talk to Supabase, and CLIs for GitHub and ClickUp. The direction the industry is heading: every service will eventually have an MCP server alongside its CLI and API. Tryps' iMessage agent (Rizwan's work) already acts as a kind of MCP — it receives structured intents from users and routes them to the right backend function.

The "aha" moment

CLIs and MCPs solve the same problem (talk to a service) for different audiences. CLIs are like giving a human a phone to call the restaurant. MCPs are like giving a robot a direct line to the kitchen's order system. Same food gets made — but the robot path is faster, more reliable, and doesn't need to interpret a human voice.

4d. RAM vs Disk vs Cache — Where Data Lives

The analogy

Think of your computer as a kitchen:

  • RAM (Random Access Memory) = the counter space. Fast to reach, limited space. Everything you're actively cooking sits here. When you close the kitchen (shut down), the counter gets wiped clean.
  • Disk (SSD/Hard Drive) = the pantry. Huge capacity, slower to access. Your recipes, ingredients inventory, everything permanent lives here. Survives a power outage.
  • CPU Cache = the spice rack right next to the stove. Tiny, but the absolute fastest thing to grab. The CPU keeps its most-used data here.
  • Cloud/Database = the restaurant supply warehouse across town. Massive capacity, but you need to drive there (network latency).

Speed comparison

TypeSpeedSizeSurvives reboot?
CPU Cache~1 nanosecond~64 MBNo
RAM~100 nanoseconds8-64 GBNo
SSD~100 microseconds256 GB - 4 TBYes
Network (DB)~1-100 millisecondsUnlimitedYes

That's a 1,000,000x speed difference between cache and network. This is why "loading" exists.

How this shows up in Tryps

When a user opens a trip, the app first checks RAM (React state — instant). If the trip isn't in RAM, it checks the local cache (AsyncStorage — fast). If not cached, it hits Supabase over the network (slow). That's why the first load of a trip takes a beat, but switching back to it is instant — it's already in RAM.

The "aha" moment

Every "loading spinner" you see in any app is the moment it's reaching from the fast counter (RAM) to the slow pantry (disk) or the distant warehouse (network). Good apps minimize this by keeping smart copies of data at each level.

4e. DNS — How "jointryps.com" Becomes a Server

What it is

DNS (Domain Name System) is the phone book of the internet. When you type "jointryps.com" into a browser, your computer doesn't know where that is. It asks a DNS server: "Hey, what's the IP address for jointryps.com?" The DNS server responds: "That's 76.76.21.21" (a Vercel server). Now your browser knows where to send the request.

The chain

  1. You type jointryps.com
  2. Your computer asks your router's DNS
  3. Router asks your ISP's DNS
  4. ISP asks a root DNS server → ".com" DNS server → Cloudflare (our DNS provider)
  5. Cloudflare says "jointryps.com = 76.76.21.21"
  6. Your browser connects to that IP address

This whole chain takes about 20-100 milliseconds. It's cached for hours, so the second visit is instant.

Why Tryps cares

Remember PR #333 (the domain consolidation)? That PR is migrating every reference from jointripful.com to jointryps.com. Both domains have DNS records pointing to the same servers, but deep links (tripful://), email "from" addresses, and API endpoints all need to match. A DNS misconfiguration means "jointryps.com" stops working for everyone, instantly.

4f. What Happens When You Run npm install

The journey

  1. Read package.json: This file lists every dependency Tryps needs (Expo, Supabase, React Navigation, etc.) — there are ~120 direct dependencies.
  2. Resolve the dependency tree: Each of those 120 packages has its own dependencies. npm figures out every package needed, recursively. Tryps ends up with ~560 packages total.
  3. Check package-lock.json: This file pins exact versions. It ensures every developer and every CI runner gets the exact same code. Without it, Nadeem might get v2.3.1 of a library while Asif gets v2.3.2, and bugs appear on one machine but not the other.
  4. Download from npm registry: npm fetches each package as a .tgz archive from registry.npmjs.org (or a cache).
  5. Extract to node_modules/: All 560 packages get extracted into a node_modules folder. This folder is typically 500MB-1GB for a React Native project. It's the heaviest thing in the repo (and it's never committed to git).
  6. Run postinstall scripts: Some packages run setup scripts after install (Expo does this to link native modules).

The "aha" moment

npm install is like ordering ingredients for a recipe. package.json is the recipe, package-lock.json is the exact brand and size of every ingredient, and node_modules is your fridge full of ingredients. You never share your fridge — you share the recipe and let everyone stock their own.

4g. OAuth — How "Sign In with Google" Works

The problem

Tryps wants to access Jake's Google Calendar (to create standup events). But Jake shouldn't give Tryps his Google password. OAuth is the protocol that solves this: "Let an app access your stuff without giving it your password."

The dance (simplified)

  1. Tryps says to Google: "Jake wants to let me see his calendar."
  2. Google shows Jake a consent screen: "Tryps wants access to your Calendar. Allow?"
  3. Jake clicks "Allow."
  4. Google gives Tryps two tokens:
    • Access token: A temporary pass (expires in ~1 hour) that lets Tryps read/write the calendar.
    • Refresh token: A permanent pass that lets Tryps get new access tokens without bothering Jake again.
  5. Tryps stores both tokens securely. Every hour, it uses the refresh token to get a fresh access token.

Why this matters

Every time Marty creates a calendar event or reads your email, he's using an OAuth access token. When you see "gog auth error" in the logs, it means the access token expired and the refresh token couldn't renew it (usually because Google revoked it for security). The fix: Jake re-authorizes via the consent screen, which issues new tokens.

The "aha" moment

OAuth is a valet key for your digital life. You're not giving Tryps the master key to your Google account — you're giving it a limited key that only opens specific doors (Calendar, Gmail) and can be revoked any time.

4h. Database Indexing — Why Some Queries Are Instant and Others Take Forever

The analogy

Imagine a library with 100,000 books. If you want "The Great Gatsby," you have two options:

  • No index: Start at shelf 1, check every book. On average, you'll check 50,000 books before finding it. (This is a "full table scan.")
  • With index: Check the card catalog, which tells you it's on shelf 47, position 12. Two lookups. Done. (This is an "index lookup.")

In Supabase/PostgreSQL

The trips table might have 10,000 rows eventually. When you query SELECT * FROM trips WHERE id = 'abc123', PostgreSQL uses the primary key index to find it in microseconds. But if you query SELECT * FROM trips WHERE destination LIKE '%Barcelona%' without an index on destination, it scans every single row.

The tradeoff

Indexes make reads fast but writes slower (every INSERT also has to update the index). It's like maintaining the card catalog — every time you add a book, you also update the catalog. Worth it for columns you search often; wasteful for columns you rarely filter by.

The "aha" moment

When the app feels slow loading trips, it's almost never the network — it's usually a missing database index causing PostgreSQL to read thousands of rows instead of jumping straight to the one it needs.

11. CS 101 — Jake's Daily Lesson

What Is a Port? (And Why "Port 3000" Keeps Coming Up)

When your computer connects to the internet, it's not just connecting to "a server" — it's connecting to a specific door on that server. These doors are called ports. There are 65,535 of them on every computer.

Think of it like an office building. The building has one street address (that's the IP address — like 178.156.176.44, Marty's server). But inside, there are thousands of office suites. Suite 80 is where the web server sits (HTTP). Suite 443 is the secure web server (HTTPS). Suite 22 is SSH (the remote login tunnel). Suite 5432 is PostgreSQL (the database). When you type https://jointryps.com, your browser goes to the building at Vercel's IP address and walks into suite 443.

Why it matters for Tryps: When Nadeem runs the Expo dev server, it starts on port 8081. When Supabase runs locally, it's on port 54321. If two services try to use the same port, one crashes — like two tenants fighting over the same office suite. When Marty's server runs OpenClaw, it listens on a port for Slack messages. That's why "port already in use" is one of the most common developer errors.

The "aha" moment: A URL like http://localhost:3000 means "talk to my own computer (localhost), door number 3000." Every running service picks a door. The internet is just millions of computers, each with thousands of numbered doors, all talking to each other.
