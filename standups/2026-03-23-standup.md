Wispr Flow is a voice-to-text tool that lets you dictate anywhere on your Mac or phone — it transcribes in real-time with near-perfect accuracy. You'll use it to answer your 5 daily questions via voice instead of typing. Jake has invited the full team.

### Claude Code Agent Teams

Agent teams let you deploy multiple Claude Code instances that work together in parallel — each one investigates a different hypothesis and they challenge each other's findings before converging on an answer. Think of it as a research swarm you can spin up from your terminal.

- Docs: [code.claude.com/docs/en/agent-teams](https://code.claude.com/docs/en/agent-teams)
- Enable it: set `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` in your Claude Code settings

**Example prompts to try:**

> "Investigate why the deep link flow drops users after OTP — spin up 3 teammates, one looking at the auth middleware, one at the mobile routing, one at the iMessage callback chain"

### Rizwan — [[agent-intelligence/objective|agent-intelligence]] (61 SC)

**1. What is your plan for designing our intelligence system and memory architecture?**

**Three Supabase memory layers** feeding two consumers (vote-on-behalf + recommendations):

- **Per-user memory** (`user_memory`)
  - Fields: signal_type, category, key, positive/negative value, weight (0.0–1.0), source
  - 7 sources across 4 engagement tiers, weights compound (capped at 1.0):
    - Vibe quiz: 0.3 · Trip intent (NL): 0.4 · Travel DNA: 0.5
    - DM conversation: 0.6 · Activity adoption: 0.5
    - Vote overrides: 0.8 · Post-trip favorites: 0.9
- **Per-trip memory** (`trip_memory`)
  - Event log per trip: votes cast, vote overrides, activities added/removed, recommendations adopted/ignored, post-trip favorites
- **Cross-trip memory** (`destination_patterns`)
  - Nightly-rebuilt aggregate of adoption and favorite rates by destination
  - Materialized from trip_memory — not a primary data source

**Context flow (4 stages):**

1. **Ingestion** — Claude function calling (recommended for SC-57, design decision pending) for free-text extraction; DB triggers for implicit signals (votes, adoptions, favorites)
2. **Storage** — Dedup and contradiction resolution: conflicting signals superseded by recency; same-key signals from multiple sources compound
3. **Retrieval** — Preference profile serialized per-request for prompt injection (proposed optimization: cache as jsonb on user_profiles)
4. **Prompt injection** — User context block in system prompt, prioritized by weight, capped under 500 tokens per memory-architecture.md

**Scaling strategies:**

- Pre-computed destination patterns (nightly rebuild, not real-time)
- Proposed: partial indexes on active-only signals, cached jsonb summaries on user_profiles
- No vector DB needed at current scale — categorical signal taxonomy is sufficient

**2. How are we going to write a successful system prompt for the agent?**

The system prompt has two layers — a static base prompt defining the agent's personality, voice, and rules, plus a dynamic user context block injected per-request from memory. The base prompt follows the voice guide from the brand design system. The dynamic block is built by `buildUserContext()` which serializes the user's top 15 preference signals with their strongest sources. For testing, I'll build an eval harness — a set of 30+ test scenarios (trip intents, vote inferences, edge cases) where we define expected behavior and score the agent's actual responses. Iteration is prompt versioning: each change gets a version tag, runs through the eval suite, and we compare scores before shipping. The key insight is that prompt quality is downstream of memory quality — if the signals going in are good, the prompt doesn't need to be clever.

**3. What research have you done so far on memory patterns, agent architectures, or system prompt strategies?**

Today I deployed three research agents that explored the t4 codebase and existing architecture docs in depth. Key findings: the NLP extraction pipeline should use Claude function calling with a defined signal schema — it gives structured, schema-validated output with the lowest complexity. Haiku handles extraction at about $0.001 per message. For recommendations, the pattern is content-based filtering on Travel DNA dimensions combined with collaborative filtering on similar groups, weighted 40/30/30 with a social graph component. The existing codebase already has dimensional scoring, group compatibility with friction flags, and contact matching — so about 60% of the recommendation infrastructure already exists. The main options I weighed were vector embeddings vs. categorical signals for memory (categorical wins at our scale) and ML ranking vs. weighted scoring function (weighted scoring wins — we don't have enough data for ML yet).

**4. What do you need from Jake or Asif to move forward?**

From Asif: we need to jointly define three interfaces before either of us implements — the DM delivery pipeline (how I trigger a batch DM through Linq), the vote override routing (how iMessage routes "switch D to yes" back to my vote engine), and a shared rate limiter so we don't spam users with overlapping proactive messages. These are SC-59, 60, and 61. I'd like to get a 30-minute session with Asif this week to spec those contracts. From Jake: I need a decision on the routing logic (SC-58) — when the agent speaks vs. stays silent. That's the hardest judgment call in the product and it needs Jake's input on at least 20 example messages before I can encode it in the system prompt.

**5. What's the one thing about building an agent intelligence layer that you think the rest of us are underestimating?**

Memory quality is harder than memory architecture. Building the tables and the pipeline is straightforward engineering. The hard part is tuning what counts as a signal vs. noise — when someone says "beaches are nice I guess," is that a preference or filler? The NLP extraction pipeline has to make that call on every message, and getting it wrong in either direction degrades the whole system. Too aggressive and we pollute memory with noise that makes bad vote inferences. Too conservative and Tier 2 and Tier 4 users feel like the agent isn't learning. I think we're also underestimating the cross-scope coordination cost with Asif's iMessage agent — three shared interfaces is a lot of surface area to keep in sync.
