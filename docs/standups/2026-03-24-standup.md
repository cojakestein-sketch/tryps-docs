**3. For the deployment guardian: what's the current EAS build process? Where are the Apple certificates and provisioning profiles — on Jake's machine, in EAS, or somewhere else? Can we run `eas build` from Hetzner, or does it need to run locally?**

The current flow is entirely local — I run it from my machine on the `production` branch. The exact steps are:

1. `git checkout production && git pull`
2. `npm install` — picks up any new or updated dependencies
3. `npx expo prebuild --clean` — regenerates the native iOS project from scratch
4. `eas build --platform ios --profile production` — sends the build to EAS cloud (~7 min). When EAS asks for Apple account, I select "no" so it auto-picks credentials from EAS config. No manual Apple ID entry.

**4. A lot of teams using OpenClaw-style agents are having Marty look at open GitHub issues, triage them, and then proactively suggest code fixes or even open PRs. Define the process: How should Marty interact with devs day-to-day? Should he monitor GitHub issues and suggest fixes? Should he comment on PRs with code suggestions? I've written a full "Marty Dev Interaction Model" doc (`docs/plans/2026-03-24-marty-dev-interaction-model.md`). The summary:

**Daily loop:** Morning brief DM to each dev (9 AM) with open PRs, assigned issues, stale work. Real-time PR review when PRs are opened. Issue triage on new issues. Evening crons (standup, nightly report, cost).

**Issue triage:** Marty auto-classifies severity, identifies the owner based on scope (iMessage stuff → me, UI → Nadeem, intelligence → Rizwan, infra → Jake), and posts a triage comment. Assigning to a dev is human-gated — Jake decides routing.

**Code suggestions:** When a P0/P1 bug is triaged and Marty can identify the root cause, it posts exact code diffs as suggestions. Opening a draft PR requires a dev thumbs-up. Marty never pushes code directly.

**PR review:** Automated on every PR using the existing tryps-pr-review skill. Convention checks, safety checks, quality checks. Approval is always human. Marty can request changes but never approves.

**Autonomous:** Reads, analysis, review comments, triage comments, informational DMs. **Human-gated:** Task creation (staged), PR merging, issue closing, dev assignment, TestFlight builds.

**5. What's the biggest risk in automating deployments? What should Marty NEVER do autonomously (e.g., submit to App Store production without approval)? What safeguards do you want?**

Biggest risk: skipping `npx expo prebuild --clean` and shipping stale native code, or building from the wrong branch. Since the app is TestFlight-only right now, the blast radius is limited to our team — but once we go to production, a bad build goes to real users.

**Marty must NEVER:** (1) Submit to App Store production — that's always manual by Jake in App Store Connect. (2) Build from any branch except `production`. (3) Skip `npm install` or `npx expo prebuild --clean`. (4) Enter Apple credentials manually during build — always select "no" and let EAS auto-pick. (5) Run `eas update` — we don't use OTA updates yet.

**Safeguards built into the tryps-deploy skill:** Pre-flight checks (typecheck + tests must pass), mandatory `npm install` and `npx expo prebuild --clean` before every build, Asif/Jake approval gate before any build+submit runs, and the skill documents the exact 6-step flow so there's no guessing.
