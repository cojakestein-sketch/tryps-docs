#### Part A: Automated QA Runner

**1. Have you used Maestro before? If not, research it today — `https://docs.maestro.dev`. Key question: can we run Maestro + Android emulator on Hetzner (Linux, no GPU)? What are the system requirements? Confirm headless mode works on a VPS.**

Haven't used Maestro before — researched it today. Yes, Maestro + Android emulator runs on Hetzner Linux with no GPU. Requirements: KVM must be available (Hetzner CCX/CPX AMD lines support it — check with `egrep -c '(vmx|svm)' /proc/cpuinfo` and `ls /dev/kvm`). Maestro itself just needs JDK 17+, installed via `curl -Ls "https://get.maestro.mobile.dev" | bash`. The Android emulator runs headless with software rendering: `emulator @avd_name -no-window -no-boot-anim -no-audio -gpu swiftshader_indirect -accel on`. Must use x86_64 system images (not ARM) for KVM acceleration. Recommended specs: 4+ vCPUs, 16 GB RAM (emulator uses 2-4 GB, builds need headroom), ~20 GB disk for SDK + emulator + system image. Ubuntu 22.04 or 24.04 LTS. Hetzner CAX (ARM Ampere) will NOT work — must be AMD-based instances.

**3. For the EAS build step: can Marty build a development client APK locally on Hetzner (`eas build --local --platform android --profile development`), or does it need to go through EAS cloud? What dependencies need to be installed (Android SDK, NDK, Java)?**

Yes, `eas build --local --platform android --profile development` works on Hetzner. No EAS cloud needed — it runs the full Gradle build locally and outputs a debug APK with the Expo dev client baked in. Dependencies:

- **Node.js** 18+ LTS
- **Java (JDK)** 17 (Expo SDK 50+ requires it)
- **Android SDK** — command-line tools, platform-tools, emulator
- **Android SDK Platform** — `platforms;android-34`
- **Android SDK Build-Tools** — `34.0.0`
- **Android NDK** — version `26.1.10909125` (exact version matters for RN native modules — check `expo doctor`)
- **CMake** — `3.22.1` (via sdkmanager, for native builds)
- **eas-cli** — `npm install -g eas-cli`

Build takes 10-30 minutes depending on hardware. The resulting APK installs directly on the emulator via `adb install`. No Android Studio required — just the SDK command-line tools. With 16 GB RAM on a CCX instance this should run cleanly.

**4. How should the QA runner integrate with the PR review pipeline? Options: (A) run on every PR automatically, (B) run nightly on develop, (C) triggered manually via Slack. What's realistic given build time?**

Hybrid approach — **(B) nightly on develop + (C) manual trigger for critical PRs.** Here's why:

- **Not (A) on every PR** — Build takes 10-30 minutes, then emulator boot + test suite adds another 5-15 minutes. That's 15-45 minutes per PR, and we'd need the emulator exclusively per run. With 3 devs pushing multiple PRs daily, this would create a queue and slow everyone down.

- **Nightly on develop (primary)** — Run the full Maestro suite every night against the latest develop branch. Catches regressions before they compound. Marty posts results to #dev by morning. This is what most teams at our scale do.

- **Manual via Slack (secondary)** — For critical PRs (anything touching auth, invite flow, navigation, expenses), a dev types `/qa-run` in Slack and Marty kicks off just the critical flows (~5 minutes). This covers the "I need to verify before merge" use case without blocking the pipeline.

- **Future (A)** — Once build caching is set up and we have a pre-built APK that only changes on native dependency updates, we can move to per-PR for the YAML-only flows (skip rebuild, just re-run tests). That brings per-PR time down to ~5 minutes.

**5. What does the Maestro MCP server give us that regular Maestro doesn't? It has 47+ tools and AI-powered assertions (`assertWithAI`, `assertNoDefectsWithAI`). Worth setting up, or start with plain YAML flows?**

**Start with plain YAML flows, add MCP server as Phase 2.**

What MCP server gives us:
- **AI-powered assertions** — `assertWithAI("the login form has email and password fields")` uses a vision model against a screenshot. No need for exact element IDs. `assertNoDefectsWithAI` checks for layout issues, truncated text, overlapping elements — bugs you didn't think to check for.
- **Dynamic testing** — AI agent decides next action based on screen state, rather than following a fixed YAML script. Good for exploratory testing.
- **Programmatic control** — 47+ tools (tap, swipe, inputText, screenshot, launchApp, setLocation, etc.) callable from any MCP client (Claude Code, Cursor, etc.).

Why YAML first:
- We already have a `.maestro/flows/` directory in t4 with partial flows for critical paths. Build on what exists.
- YAML flows are deterministic — same test, same result. Better for CI/CD regression testing.
- MCP server requires a running emulator AND an MCP client — more moving parts to debug on Hetzner.
- The AI assertions cost money per call (vision model inference).

Phase 2 plan: Once YAML flows are stable and running nightly, add MCP server for exploratory testing and visual regression checks. Setup is just `npx -y @mobile-dev-inc/maestro-mcp` in the MCP client config.

**6. After reading the entire memory system, what's your assessment? Is the memory architecture sound — curated long-term (MEMORY.md) + daily ephemeral (memory/*.md) + distillation skill? What's missing? How should memories be structured, tagged, or indexed for better retrieval?**

The three-tier architecture is sound — it's the right pattern. Curated long-term (MEMORY.md, 22KB) + daily ephemeral (memory/*.md, 16 files) + distillation skill (memory-distill) is a good foundation. The boot sequence in AGENTS.md (read SOUL → USER → TOOLS → last 3 days memory → MEMORY.md for main sessions only) is well-designed. The standup-driven feedback loop (answers → distill → next day's questions) is the strongest part.

**What's missing:**

1. **No strict daily memory template** — Format varies wildly across daily files. 2026-03-08 is a brain dump with personal interests, 2026-03-11 is a session log with dialogue, 2026-03-23 is research notes. The memory-distill SKILL.md defines a template (Key Facts, Decisions, Blockers, Promises, Surprises, Contradictions) but it's not consistently applied.

2. **No memory sections or tagging in MEMORY.md** — It's one long document with headers but no way to query "what does Marty know about expenses?" or "what are current blockers?" Adding frontmatter tags (e.g., `topics: [expenses, auth, team]`) to daily files would help retrieval.

3. **No semantic indexing** — When Marty boots, he reads the full 22KB MEMORY.md every time. At current scale this is fine. At 50KB+ it'll eat context window. Proposal: split MEMORY.md into topic-indexed sections (team.md, product.md, conventions.md, gotchas.md) with an index file.

4. **MEMORY.md security boundary is good** — Only loaded in main Jake sessions, not group chats. This is correct.

5. **Missing USER.md and TOOLS.md inconsistency** — AGENTS.md references both in boot sequence. TOOLS.md exists (300+ lines, great). USER.md exists but could be richer.

**7. Marty's memory currently has no expiry or relevance scoring. Old daily memories just accumulate. How should we handle memory decay? Should Marty auto-archive stale memories? What signals indicate a memory is no longer useful?**

Yes, we need memory decay. Here's the proposed system:

**Three-tier decay:**

1. **Daily files (memory/*.md)** — Active window: last 7 days. After 7 days, auto-archive to `memory/archive/`. Marty only reads last 3 days on boot (already in AGENTS.md), so files older than 7 days are never read anyway. Keep them in archive for human reference, not agent consumption.

2. **MEMORY.md sections** — Add `last_verified: YYYY-MM-DD` to each major section. During weekly heartbeat (Friday 5pm, already scheduled), Marty scans each section and flags anything not verified in 30+ days. Stale sections get moved to a `## Archive` section at the bottom of MEMORY.md, or pruned entirely if superseded.

3. **Promise tracking** — Promises with target dates that pass without resolution should auto-surface in standup questions for 3 consecutive days, then get marked as "stale/unresolved" and archived.

**Staleness signals:**
- Daily file older than 7 days → archive
- MEMORY.md section not referenced or verified in 30+ days → flag for review
- Promise past target date by 7+ days → escalate then archive
- Blocker resolved (PR merged, task closed in ClickUp) → remove from active blockers
- Team member info contradicted by newer info → supersede old entry

**Implementation:** Add a `memory-cleanup` cron job (weekly, Sunday night) that moves old daily files to archive and generates a staleness report for MEMORY.md sections. Marty reviews the report during Monday's heartbeat and prunes.

**8. How does deer-flow (ByteDance's open-source agent framework — see Tools section above) handle agent memory? Compare their memory/skill/subagent patterns to what we have in OpenClaw. What can we steal?**

Researched deer-flow (39.3k stars). Key finding: **deer-flow has essentially no memory system.** It's a stateless research pipeline built on LangGraph — each run starts fresh. Memory is just an in-memory Python state object that dies when the run ends. No cross-session persistence, no memory decay, no indexing, no retrieval.

**Comparison:**

| Dimension | deer-flow | OpenClaw (Marty) |
|-----------|-----------|-----------------|
| Persistence | None — ephemeral per run | Full — markdown files survive across sessions |
| Cross-session | No | Yes — boot sequence reads last 3 days + MEMORY.md |
| Storage | In-memory Python dicts | Human-readable .md files on disk |
| Searchability | Current context only | Grep/glob across all files |
| Human-editable | No (opaque state) | Yes — anyone can read/edit memory files |
| Version control | No | Git-native (tryps-docs repo) |
| Decay/relevance | None | None yet (proposed above in Q7) |

**Where deer-flow excels instead:** Multi-agent orchestration (supervisor → planner → researchers → reporter), LangGraph state-graph routing, and tool-use coordination. These are orthogonal to memory.

**What we can steal:**
1. **Supervisor/coordinator pattern** — deer-flow's top-level agent that decomposes tasks and delegates to specialists is cleaner than our current flat skill invocation. We could add a "Marty dispatcher" that routes requests to the right skill automatically instead of Jake invoking skills manually.
2. **Research workflow decomposition** — Their plan → search → analyze → synthesize pipeline is a good pattern for Marty's nightly report and topic suggestion skills.
3. **Nothing on memory** — Our markdown-file approach is significantly more capable. deer-flow is the "no memory" baseline — any system that remembers anything across sessions outperforms it.

Bottom line: deer-flow validates that our memory architecture is ahead of even well-funded open-source agent frameworks. The gap is in decay/cleanup, not in the core pattern.
