### Rizwan — agent-intelligence (61 SC)

**1. PR #303 (agent intelligence memory system) merged on March 25. Walk me through the architecture — what got shipped? Per-user memory, per-trip memory, cross-trip learning, or some subset?**

PR #303 shipped the full memory foundation — all three layers. Per-user memory (`user_memory` table) for preference signals with weighted signal types (preference, behavior, interaction_style, correction) and source tracking. Per-trip memory (`trip_memory` table) for trip-specific events like votes, overrides, activities added/removed, and recommendations adopted/ignored. Cross-trip learning via `destination_patterns` table with aggregated anonymized patterns per destination, rebuilt via nightly cron. Also shipped vote-on-behalf and recommend-activities as edge functions, the NLP extraction pipeline (`extract-memory-signals` via GPT-4o-mini), DNA/vibe sync functions, 3 DB migrations, 5 DB triggers, and 1000+ seed activities across global destinations.

**2. The memory wiring into agent-router had a revert + re-apply (#304). You added buildUserContext + fireAndForgetExtraction. What does "fire and forget" mean here — if extraction fails silently, how do you know the memory system is actually learning?**

The revert happened because the first wiring had no safety guards — if extraction threw, it killed the chat response. Re-applied with try/catch so it degrades gracefully. Fire-and-forget means we don't await the extraction fetch — user gets their reply instantly, extraction runs in the background. We know it's learning via the `signal_extraction_log` table which audits every extraction call with user_id, signals_extracted count, low_confidence_skipped count, latency_ms, and input_hash (no raw text per SC-55). Can also query `user_memory` directly to verify signals are accumulating. Honest gap: no retry or alerting if the fetch itself fails at the network level — that's a monitoring improvement I need to add.

**3. PR #299 (font scaling protection) has been open since March 26 and isn't merged yet. What's blocking it? Is this related to agent-intelligence scope or a side fix?**

PR #299 is a side fix — font scaling protection for home and activity components. Not related to agent-intelligence scope at all. It's a small PR (45 additions, 32 deletions) standardizing text scaling behavior. No failing checks, just awaiting human review. I'll nudge someone to merge it.

**4. The spec mentions three sub-systems: memory architecture, vote-on-behalf, and recommendations engine. Memory is landing now — what's the order for the other two? Have you started on vote-on-behalf at all?**

All three sub-systems shipped edge function code in PR #303 — memory architecture is most complete (12/17 SCs ready), vote-on-behalf is mid-progress (7/15 SCs ready), and recommendations engine is early (2/18 SCs ready).
